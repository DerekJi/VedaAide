## 二、测试架构与 Telegram Bypass 策略

### 2.1 核心问题

Telegram 消息处理逻辑（`process_user_message`、`confirm_data`）原本内嵌在 aiogram 的 handler 中，依赖 `types.Message`、`FSMContext`、`CallbackQuery` 等 Telegram 专有对象。这使得集成测试无法绕开 Telegram 平台直接触发业务流程。

### 2.2 解决方案：提取 `MessageProcessor` 层

将核心业务逻辑提取为独立的 `MessageProcessor` 类，**不依赖任何 aiogram/Telegram 对象**：

```
bot_app/
  message_processor.py     ← 新增：纯业务逻辑层
  main.py                  ← 瘦化：只做 Telegram 消息格式转换 + 调用 MessageProcessor
```

**MessageProcessor 接口**：

```python
class MessageProcessor:
    async def process(text: str) -> ProcessResult
    # 返回: { skill_name, extracted_data, needs_confirmation }

    async def confirm(skill_name: str, extracted_data: dict, raw_text: str) -> bool
    # 写入数据库，返回是否成功

    async def get_todays_events() -> list
    async def generate_daily_reminder() -> str
    async def generate_weekly_preview() -> str
```

**好处**：
- 集成测试直接调用 `MessageProcessor`，完全不需要 Telegram
- aiogram handler 变为薄包装层，逻辑清晰
- `MessageProcessor` 可注入 Mock 的 `OllamaClient`，方便集成测试

### 2.3 测试分层

```
┌──────────────────────────────────────────────┐
│  E2E 测试（端到端，模拟真实对话流）           │  ← 少量、覆盖核心场景
├──────────────────────────────────────────────┤
│  集成测试（MessageProcessor + 真实 DB）       │  ← 中量，跳过 Telegram
├──────────────────────────────────────────────┤
│  单元测试（单个模块，全部 Mock 外部依赖）     │  ← 大量、快速
└──────────────────────────────────────────────┘
```

| 层级 | 依赖 | 覆盖目标 | 执行频率 |
|------|------|----------|----------|
| 单元测试 | 无外部服务 | 数据解析、路由逻辑、规则推算 | 每次提交 |
| 集成测试 | 真实 SQLite、Mock Ollama、`MessageProcessor` | 数据读写流程、Skill 执行链 | 每次提交 |
| E2E 测试 | 真实 SQLite、真实 Ollama | 完整对话场景 | 按需/定时 |

### 2.4 测试中的数据库验证

集成测试中，有两种验证方式可组合使用：

| 方式 | 适用场景 | 说明 |
|------|----------|------|
| 直接查询 DB（`DatabaseClient`） | 验证写入是否正确 | 最直接，不依赖 LLM |
| 通过 `MessageProcessor` 查询 | 验证端到端查询功能 | 同时验证查询逻辑 |

测试用 DB 使用 `tmpdir` 临时路径，每个测试函数独立，互不干扰。

### UT-07：事件频率自动推断（Pattern Detection）

**被测对象**：频率分析模块中的间隔计算和规律判断逻辑（纯函数，不依赖 LLM 和 Telegram）  
**隔离方式**：使用固定历史事件列表作为输入，测试间隔计算和置信度判断

| 用例 ID | 输入：历史记录 | 期望输出 | 说明 |
|---------|--------------|----------|------|
| UT-07-01 | 同 item 记录 2 条，间隔精确 90 天 | interval=90天, confidence=高 | 最简情况 |
| UT-07-02 | 同 item 记录 3 条，间隔分别为 85、95 天 | interval≈90天（平均/中位数） | 波动小，规律成立 |
| UT-07-03 | 同 item 仅 1 条记录 | 返回 None 或 "数据不足" | 无法推断 |
| UT-07-04 | 同 item 记录 3 条，间隔分别为 10、90、300 天 | 返回 None 或 confidence=低 | 波动超过 50%，不生成规律 |
| UT-07-05 | 已知 interval=90天，上次时间=60天前 | 返回"未到期，还有约 30 天" | 按需查询回答 |
| UT-07-06 | 已知 interval=90天，上次时间=100天前 | 返回"已超期 10 天，该换了" | 到期提醒 |
| UT-07-07 | interval=90天，无上次记录 | 返回"没有找到上次记录" | 边界 |

**设计备注（LLM 成本取舍）**：
- **按需分析**：用户主动提问时触发 LLM 分析历史记录，结果缓存到 `background_rules`；下次查询优先读缓存，避免重复分析。成本低，首次查询略慢。
- **定时批量分析**：每周报告生成时，对最近 N 条 `life_events` 做批量频率分析，更新 `background_rules`；主动贴心，但每周多消耗一次 LLM 调用。
- **推荐策略**：两者结合——每周批量分析作为后台任务，按需查询作为兜底；`background_rules` 表同时作为缓存层，间隔计算的数学部分（均值/中位数/标准差）不走 LLM，LLM 仅负责"是否是同一类事件"的语义判断。

## 五、端到端测试场景（E2E）

> 使用真实 Ollama 服务（本地运行），模拟完整用户对话。不依赖 Telegram 平台，通过直接调用 Bot 逻辑层实现。

```

期望: user_profiles 写入 son_name, son_age（或相关字段）

期望: scheduled_events 写入 5 条记录，recurrence_rule 各自正确

触发"今天是周一，生成提醒"：
期望: 提醒内容包含游泳（游泳包）和 Scouts

触发"今天是周四，生成提醒"：
期望: 提醒内容包含篮球和音乐剧排练

### E2E-03：camping 行程录入与提前预警

```
用户（邮件转发）: "下周六学校有 camping，需要帐篷、睡袋、头灯"

```

期望: life_events 再写入一条，同 item

触发频率分析（按需或定时）：
期望: background_rules 写入 freq:牙刷头，interval≈90天，last_date=2026-04-01

用户（2026-06-15）: "是不是该换牙刷头了？"
// 距上次 2026-04-01 已 75 天，未到 90 天
期望: "还有约 15 天，上次 4 月 1 日换的，通常每 90 天换一次"

用户（2026-07-10）: "是不是该换牙刷头了？"
// 距上次已 100 天
期望: "是的，已经超过约 10 天了，该换了"

用户（2026-07-10）: "今天换了牙刷头"
期望: life_events 写入；background_rules 的 last_date 更新为 2026-07-10，interval 用新的 3 条记录重新计算

触发周报生成时的主动分析：
期望: 如当前日期距 last_date 接近 90 天，周报中包含"牙刷头快该换了"提示
```

## 八、测试数据规范

### 8.1 标准测试语料库

以下语料覆盖所有核心意图，测试时应全部通过：

**生活事件类（record_event）**
- "今天剪头了"
- "在 Woolworths 买了 2kg 牛肉"
- "今天吃了 5 个饺子"
- "Marco 今天吃了 10 个饺子，我吃了 8 个"
- "冰箱里还有半棵白菜"

**计划事件类（schedule_event）**
- "孩子每周二有游泳课"
- "下周四下午三点孩子看牙医"
- "三月二十号全家去camping，要带帐篷睡袋头灯"
- "每周一早上有英语课"

**背景信息类（update_profile / background_rule）**
- "我儿子叫 Marco，今年 10 岁"
- "我家住在 Sydney Chatswood"
- "绿桶黄桶每两周交替，本周（2026-03-09这周）是黄桶"
- "我们家三口人：我、我老婆、儿子 Marco"

**查询类（query）**
- "今天该推哪个桶？"
- "上次买猪肉是什么时候？"
- "下周有什么安排？"
- "Marco 下周有几次篮球？"
- "冰箱里还有什么肉？"

### 8.2 边界语料

- 非常简短的输入："到了"、"买了"、"好的"
- 带噪音的输入："额那个，嗯，今天大概买了差不多两斤肉吧"
- 多意图混合："今天剪头了，顺便买了菜，下周Marco有游泳"
- 纯问候："你好"、"谢谢"
